# 内容分析报告

## 一、核心内容
### 1. 核心论点
传统 `<script src>` 会阻塞解析与后续资源，导致 DOM 未就绪与串行加载；应把脚本置底、使用 defer/async、动态创建 script 或按需异步加载以改善首屏。

### 2. 关键概念
| 概念 | 文中定义/内涵 |
|------|--------------|
| 阻塞执行 | JS 执行时阻塞渲染与其它下载 |
| 串行载入 | 多 script 标签顺序下载执行 |
| defer | 延迟到文档解析后、DOMContentLoaded 前按序执行 |
| async | 下载并行、下载完即执行，顺序不定 |

### 3. 文章结构
淘宝内网竞赛引子 → 两大特性 → 传统 head script 阻塞示例一 → document.write 仍阻塞示例二 → defer/async 语义 → 动态 DOM 创建 → 按需加载与更多资源 → 链到渲染文。

### 4. 案例与证据
- `async_test01.html` 单 alert 阻塞演示。
- `loadjs` + document.write 顺序反直觉（示例二）。
- 操作后续 DOM 报错因 HTML 未解析。
- 推荐脚本放页面底部或 onload/DOMReady。

## 二、背景语境
### 1. 作者身份
陈皓，酷壳；前端性能与浏览器行为科普，承《浏览器渲染原理》。

2013 年 Web 性能竞赛；开发者混淆“异步”与“非阻塞”。

### 3. 写作意图
纠正装载模型误解；提供可测 HTML 示例；推动 defer/async 最佳实践。

### 4. 底层假设
经典浏览器 HTML 解析器；讨论 ES5 时代；读者写多 script 页面。

## 三、批判性审视
### 1. 主要反对意见
HTTP/2 多路复用、module、preload 改变图景；inline 关键 JS 仍必要；现代框架打包自行拆分。

### 2. 论证漏洞
未量化 defer vs bottom；document.write 在 2013 后更被禁用；CSP 未谈。

### 3. 适用边界
多外链脚本落地页；SPA 由打包器管理；Service Worker 另论。

### 4. 回避问题
type=module；importmap；性能指标 FCP/LCP 术语当时未普及。

## 四、价值提取
### 1. 可复用思考框架
每个 script 问：是否必须阻塞？能否 defer/async？能否动态按需？

### 2. 对从业者的启发
默认 defer 外链；避免 document.write 载脚本；测 Performance 面板。

### 3. 对决策者的启发
性能规范写入前端 checklist；内网竞赛考基础模型仍有价值。

### 4. 认知升级点
“下载完”≠“不阻塞解析”；DOM 操作时机依赖装载顺序。

## 五、观点辩论

### 核心观点列表（≤5）
1. 默认 script 会阻塞 HTML 解析与渲染。
2. document.write 插入 script 不能解决阻塞。
3. defer/async 是控制装载时机的标准手段。
4. 动态创建 script 可实现真正异步加载。
5. 脚本置底是简单有效的传统优化。

### 观点一深度辩论
**同步 script 阻塞渲染是浏览器默认且影响首屏的主因之一。**

**第一轮**
- 支持方：示例一与 DOM 未生成报错；与渲染文 reflow 链一致。
- 反对方：async CSS 也阻塞渲染；inline script 无网络但仍阻塞。
- 综合方：外链同步 script 兼具网络与执行阻塞 | 共识进度：🟢

**辩论结论**：共识表述——未标记 defer/async 的外链脚本会阻塞解析，首屏优化必须显式管理其位置与属性。

### 观点二深度辩论
**document.write 加载脚本仍阻塞页面。**

**第一轮**
- 支持方：示例二 alert 顺序证明；HTML5 解析器对 write 敏感。
- 反对方：已废弃用于 SPA；历史文章价值降低。
- 综合方：历史反模式，今应避免 | 共识进度：🟢

**辩论结论**：共识表述——document.write 注入 script 不是异步方案，现代开发应禁用该模式。
