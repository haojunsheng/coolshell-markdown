# 内容分析报告

> **原文**：`2016/性能测试应该怎么做？.md`  
> **作者**：陈皓（左耳朵耗子）

## 一、核心内容

### 1. 核心论点

一份“只报平均值、只在低速测延迟、不看成功率”的性能报告**会把人带进沟里**；严谨性能工程要把 **吞吐、延迟分位、资源利用、成功率、稳定性** 绑在一起测，并以 SLA 语言（如 TP99）约束系统可承受的极限负载。

### 2. 关键概念

| 概念 | 文中定义/内涵 |
|------|----------------|
| 平均值陷阱 | 少数极慢样本会严重扭曲均值；体育评分常去头尾。 |
| TP/百分位 | TP50/TP90 等表示“百分之多少请求低于某阈值”。 |
| 吞吐与延迟耦合 | 负载升高系统不稳时，孤立谈 QPS 无意义；需用延迟卡吞吐上限。 |
| Soak/Burst Test | 长时间浸泡与极限/脉冲交替，观察泄漏与恢复。 |
| 软/硬报警线 | 路透经验：以峰值吞吐的 66.7%、80% 等设运营阈值。 |

### 3. 文章结构

由 Dubbo 公开报告起笔，列三条主要批评。分章论证：为什么均值不靠谱（引用 Dynatrace 文章、举排序例子、过渡到百分位）；为何延迟必须跟吞吐挂钩（配曲线图说明抖动与天花板）；为何必须看成功率。最后给出“如何严谨做”的六步流程（定义 TP99+成功率→找最大稳态吞吐→Soak→极限→Burst→低吞吐/小包），附路透报警系数与结语强调工程科学态度。

### 4. 案例与证据

- 自拟数组 `[10ms,1s,200ms,100ms]` 手算 TP50/TP90。  
- 路透“99.9%<1ms 且均值<1ms”的工业要求作锚点。  
- Benchmark 曲线图说明吞吐与稳定性关系。

## 二、背景语境

### 1. 作者身份

陈皓具 Thomson Reuters 实时金融网与高并发优化经历，文中“老老东家”为亲身语境；批评指向业内常见PR式测试报告。

### 2. 写作背景

2016 年微服务与中间件营销盛行，“性能报告”常被销售化；工程师需要**抗噪方法论**。

### 3. 写作意图

给出一套可检查的清单，帮助读者识别不靠谱 benchmark；推广把 SLA 当成合同语言而非公关辞令。

### 4. 底层假设

假设目标系统是线上长期运行、失败成本高的服务；假设团队有能力投入长时间压测与观测。

## 三、批判性审视

### 1. 主要反对意见

快速迭代团队可能先做 smoke perf 再补全量；成本有限时难以跑满 7 天 Soak。也有人认为不同业务（批处理 vs 在线）应有不同指标权重。

### 2. 论证漏洞

正文将“中位数”英文写成 *Mean*，属术语笔误（mean 为均值，median 为中位数），读者若照抄会在专业场合露怯。对 Dubbo 原文未逐条引用细节，批评停留在模板层。极端金融场景阈值泛化到所有 Web 服务可能过严。

### 3. 适用边界

初创验证期可简化，但**发布前**应补齐核心耦合测试；读多写少与写多读少系统的热点不同，需调配方。

### 4. 回避问题

未展开分布式压测协调、仿真生产流量回放、代价与风险（压测拖垮共享环境）；对 CI 中的轻量性能回归着墨少。

## 四、价值提取

### 1. 可复用思考框架

写测试计划时先写**SLO 句子**：“在 TP99<X ms 且错误率<Y% 时，系统最大可持续 QPS 为 Z”；然后设计阶梯加压找到 Z；再做时间与突发维度.validate。

### 2. 对从业者的启发

报告里看见只有 avg 就要求补直方图或分位；把成功率曲线与延迟放在同一 dashboard；学会用生产尺寸混合 payload。

### 3. 对决策者的启发

为性能团队买时间与机器；把 SLA 写进合同并配纠错预算；别用短期营销活动透支架构余量。

### 4. 认知升级点

从“能压到多少 QPS”升级到“**在什么用户体验底线之下**能压到多少 QPS”。

## 五、观点辩论

### 核心观点列表（≤5）

1. 平均值不足以描述延迟分布。  
2. 延迟必须与吞吐联合扫描，否则指标无意义。  
3. 成功率是性能指标硬约束而非可选项。  
4. 严谨流程包含浸泡、极限与突发等时序维度。  
5. 性能是科学工程，不是拍脑袋写漂亮数。

> 筛选理由：对齐原文批评三件套+六步法的递进逻辑。

### 观点一深度辩论

**[是否应彻底抛弃均值]**

**第一轮**  
- 支持方：尾部风险由分位刻画更准。  
- 反对方：均值在容量规划与某些聚合成本模型仍有位置。  
- 综合方：**分位为主、均值为辅**。 | 共识进度：🟢  

**辩论结论**：文章反的是“唯均值”，不是反算术平均本身。

### 观点二深度辩论

**[7×24 Soak 是否过重]**

**第一轮**  
- 支持方：内存泄漏与慢增长问题只能靠时间显形。  
- 反对方：云原生弹性与快速回滚可换风险模型。  
- 综合方：**按业务后果分级**，核心链路长跑，边缘可缩短。 | 共识进度：🟢  

**辩论结论**：原则坚持，时长可 scaled。

### 观点三深度辩论

**[高标 SLA 会否压垮创新速度]**

**第一轮**  
- 支持方：事前明确可避免上线后天价救火。  
- 反对方：过早优化拖慢产品实验。  
- 综合方：**分层 SLO**：实验环境宽容，核心链严格。 | 共识进度：🟢  

**辩论结论**：把严标准画在正确边界上。

### 观点四深度辩论

**[公开性能报告可信度谁监督]**

**第一轮**  
- 支持方：应用本文检查清单即可群众监督。  
- 反对方：造假仍可 selective disclosure。  
- 综合方：倡导**可复现脚本与数据集脱敏开放**。 | 共识进度：🟡  

**第二轮**  
- 支持方：开源压测场景成为社区规范后可比性上升。  
- 反对方：商业机密限制全开。  
- 综合方：公开方法论+关键汇总图是折中。 | 共识进度：🟢  

**辩论结论**：透明应追求，落到商业可行梯度。

### 观点五深度辩论

**[资源利用率指标是否与用户感知同等重要]**

**第一轮**  
- 支持方：饱和资源预示即将崩塌。  
- 反对方：用户只关心体验与价格。  
- 综合方：利用率是**先行指标**，需映射到 SLO。 | 共识进度：🟢  

**辩论结论**：两类指标配套，不做二选一。
