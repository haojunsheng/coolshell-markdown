# 内容分析报告

## 一、核心内容

### 1. 核心论点
IE6/IE7 存在已被公开利用代码的 CSS 处理 0day 漏洞，在微软补丁发布前应升级或换用 Firefox/Chrome，并警惕攻击代码迅速武器化。

### 2. 关键概念

| 概念 | 文中定义/内涵 |
|------|--------------|
| 0day | Symantec 已披露、尚无官方补丁时已被利用的漏洞 |
| CSS STYLE outerHTML 触发 | 通过 `getElementsByTagName("STYLE")[0].outerHTML="1"` 等与 CSS 解析相关的路径触发 |
| 武器化曲线 | 当前 PoC 不可靠，但专家预期周末内将出现可靠 shellcode 并大规模挂马 |

### 3. 文章结构
报道 Symantec 与 Bugtraq 信息源；关联本站早前 IE+CSS BUG 文；建议升级 IE8 或换浏览器；解释攻击可靠性现状与 XP SP3 已验证；贴出含 shellcode 的 HTML/JS 利用代码；批评微软补丁响应节奏滞后。

### 4. 案例与证据
- Symantec 博客链接（2009-11-21）
- Bugtraq 邮件列表公开 exploit
- 本站《IE的CSS相关的BUG》交叉引用
- 完整攻击页面代码块（含 unescape shellcode 与内存喷射模式）

## 二、背景语境

### 1. 作者身份
陈皓，酷壳；本文具安全预警性质，面向仍滞留 IE6/IE7 的中文开发者与企业用户。

### 2. 写作背景
2009 年末 IE6/7 仍大量存在于中国与全球；浏览器 CSS 实现复杂、历史债务多；微软补丁周期常遭社区批评。

### 3. 写作意图
填补中文社区信息空白、推动立即升级、预告更大规模攻击；同时建立酷壳在安全快讯上的存在感。

### 4. 底层假设
- 读者能自行更换浏览器或禁用脚本
- 公开 exploit 代码不会显著降低攻击门槛（已公开则风险已定）
- 微软将再次「慢半拍」
- IE8 相对安全（作者个人建议仍带保留）

## 三、批判性审视

### 1. 主要反对意见
- 责任披露：全文转载 exploit 可能加速滥用，与「预警」目标冲突
- 企业 IT：大量内网应用绑定 IE6，无法「一键升级」
- 技术准确性：IE8 是否完全不受影响需版本与补丁级别验证，文未给出 CVE 编号

### 2. 论证漏洞
- 「中文社区尚无报道」难以核实且有时效性短
- 对中招规模纯推测（「相信周末有人改进」）
- 未提供临时缓解措施（禁用 JS、EMET、XSS 过滤等）的优先级清单

### 3. 适用边界
**成立**：2009 年 IE6/7+XP 环境、已公开 PoC 的紧急响应。**失效**：现代 Chromium 内核、已终止 IE 支持的环境（历史档案价值为主）。

### 4. 回避问题
- 企业迁移成本与国产化/ActiveX 依赖
- 微软协调披露时间线与 CVE 细节
- 攻击代码法律与伦理（转载边界）

## 四、价值提取

### 1. 可复用思考框架
**0day 时间线模型**：披露 → 不可靠 PoC → 可靠武器化 → 挂马潮 → 补丁。**浏览器风险分层**：版本×操作系统×脚本开关。

### 2. 对从业者的启发
安全预警应附 CVE、影响版本、缓解措施、是否公开 PoC 的风险评估；内网系统应维护「IE 依赖清单」与迁移路线图，而非反复救火。

### 3. 对决策者的启发
遗留浏览器是组织级风险敞口；补丁 SLA 需与暴露面匹配；关键业务不能无限期绑定过时引擎。

### 4. 认知升级点
CSS 解析与 DOM 操作交界同样可 RCE，非只有 ActiveX；「升级 IE8」在当年是过渡方案，长期解法是淘汰 Trident。

## 五、观点辩论

### 核心观点列表（≤5）
1. IE6/7 用户应立即停止使用并换浏览器
2. 当前 PoC 不可靠但很快会可靠化
3. 微软补丁响应一贯滞后
4. 公开 exploit 代码有助于中文社区知情
5. 漏洞与 IE 处理 CSS 的历史缺陷一脉相承

### 观点一深度辩论
**IE6/7 用户应立即停止使用并换浏览器**

**第一轮**
- 支持方：① 0day 无补丁；② 换浏览器成本低于被控；③ Symantec 已证实。
- 反对方：① 企业内网无法换；② 可禁用 JS 降级风险；③ IE8 未完全验证。
- 综合方：对个人用户**换浏览器是最优**；对企业需「隔离+缓解+排期迁移」组合。 | 共识进度：🟢

**辩论结论**
- 共识表述：消费级与可控环境应立即弃用 IE6/7；企业环境需配套兼容方案而非简单口号。

### 观点二深度辩论
**当前 PoC 不可靠但很快会可靠化**

**第一轮**
- 支持方：历史规律支持武器化加速；Symantec 专家判断。
- 反对方：也可能被微软 0day 采购或静默修复打断；预测不可证。
- 综合方：应作**高风险情景规划**，而非确定性预言。 | 共识进度：🟢

**辩论结论**
- 共识表述：将「可能快速可靠化」作为应急假设合理，应急资源应按此准备。

### 观点三深度辩论
**微软补丁响应一贯滞后**

**第一轮**
- 支持方：IE 时代多次先例；周末攻击窗口存在。
- 反对方：复杂漏洞需回归测试；近年 MSRC 流程已改进（后验）。
- 综合方：在 2009 语境下批评有依据；不宜泛化为今日微软全貌。 | 共识进度：🟢

**辩论结论**
- 分歧定性：适用条件为 2009 年 IE 补丁文化；对现代 Windows 更新机制需另论。

### 观点四深度辩论
**公开 exploit 代码有助于中文社区知情**

**第一轮**
- 支持方：透明利于防御与检测规则编写。
- 反对方：脚本小子复制粘贴加速伤害；应只描述原理不放 shellcode。
- 综合方：防御方需要 IoC/行为特征，但**全文转载 shellcode** 弊大于利；可链 Bugtraq 外站。 | 共识进度：🟡

**第二轮**
- 支持方：当年社区习惯全文转载技术细节。
- 反对方：现代负责任披露强调最小化利用代码。
- 综合方：知情与伤害权衡下，**链接+缓解**优于全文复制。 | 共识进度：🟢

**辩论结论**
- 观点修正：预警价值成立，但发布形式应改为摘要+外链，避免直接降低攻击门槛。

### 观点五深度辩论
**漏洞与 IE 处理 CSS 的历史缺陷一脉相承**

**第一轮**
- 支持方：文内自引 CSS BUG 文；STYLE 操作与历史问题一致。
- 反对方：0day 可能是新代码路径，需 CVE 级分析才能说「一脉相承」。
- 综合方：属**合理假设**待技术分析证实。 | 共识进度：🟢

**辩论结论**
- 共识表述：从攻击向量看与 CSS/DOM 交界问题同类，精确归因需官方安全公告。
