## 三、批判性审视
### 1. 主要反对意见
未讨论 OAuth、配额、数据合规；Google API 已多轮变更，链接或失效；Spreadsheet 非正式数据库。

### 2. 论证漏洞
无代码片段；未对比自建 Grafana/内部 portal；「最方便」未论证。

### 3. 适用边界
成立：小团队快速共享指标、原型 dashboard。失效：强合规、离线、大规模实时分析。

### 4. 回避问题
认证安全、SLA、成本、与 App Engine 等其它 Google 产品关系。

## 四、价值提取
### 1. 可复用思考框架
「先汇集、再发布」：数据管道与展示层分离，展示层可选用 SaaS 降运维。

### 2. 对从业者的启发
脚本化重复粘贴；评估云表格 API 前先看数据分级与留存政策。

### 3. 对决策者的启发
小团队可用托管表格加速 MVP，成长后迁移到正式 BI 平台。

### 4. 认知升级点
2009 年已在实践「用 API 把后台数据推到人人可看的 URL」，是今日 data portal 的前身。

## 补充：两阶段 pipeline 的工程细节
第一阶段「汇集到 CSV/数据库」应明确刷新频率、去重键与数据质量校验，否则 dashboard 展示垃圾。第二阶段「生成在线图表」在 2009 年常指服务端渲染图片或 Flash 图表；Spreadsheet 折中方案让人类直接筛选透视，适合运营与非技术干系人。Matt Cutts 作为 Google 员工示例，暗示当时 Google Data API 文档与配额政策；现代读者应查阅 Google Sheets API v4、服务账号与 OAuth 2.0，而非复制 2009 年认证方式。若数据含个人信息，还需评估是否满足最小披露原则，表格共享链接的权限模型（仅组织内/公开）必须在设计首日确定。
