# 内容分析报告

## 一、核心内容
### 1. 核心论点
HTTP 动词属于控制逻辑，应区分业务语义：GET/PUT/DELETE 等幂等与安全属性不同；全用 POST 破坏缓存、幂等、协作与架构原则，「HTTPS 仅 POST 安全」是对协议的误解。

### 2. 关键概念
| 概念 | 文中定义/内涵 |
|------|--------------|
| 业务逻辑 vs 控制逻辑 | Method 在协议头，属控制面 |
| 幂等性 | RFC7231，GET/PUT/DELETE 幂等，POST 非幂等 |
| RESTful | 资源导向 + 标准 HTTP 语义 |
| 原则五 | 《架构原则》中遵循标准与最佳实践 |

### 3. 文章结构
V2EX 引子（同事全 POST）→ 为何要不同动词 → Restful 复杂查询 → 回应六点（安全、沟通、早回家、优雅等）。驳论+规范教育。

### 4. 案例与证据
- V2EX 帖与 Twitter 嘲讽「钉枪装修」类比
- HTTP 方法表（幂等、安全列）
- 链至 2021《架构原则》反对全 200 返回码

## 二、背景语境
### 1. 作者身份
陈皓，架构原则系列作者，强烈工程规范偏好。

### 2. 写作背景
2022 年 2 月国内 API 设计争论；「能跑就行」文化。

### 3. 写作意图
正视听；给可反驳「全 POST」的论据；关联团队长期成本。

### 4. 底层假设
团队会长期维护 API；网关/CDN/监控依赖 Method 语义。

## 三、批判性审视
### 1. 主要反对意见
GraphQL 单端点；部分 RPC-over-HTTP 只用 POST 亦成功；内部系统可牺牲 REST 纯度。

### 2. 论证漏洞
对 gRPC/GraphQL 例外讨论不足；「优雅能当饭吃」反驳略情绪化。

### 3. 适用边界
对外公开 API、多团队集成；封闭内部可约定但需文档。

### 4. 回避问题
Idempotency-Key 等 POST 幂等补丁实践未展开。

## 四、价值提取
### 1. 可复用思考框架
API 设计先画资源与动词，再讨论 URL；用 RFC 表做 review checklist。

### 2. 对从业者的启发
GET 可缓存、DELETE 幂等重试安全；设计评审拒绝「一个 POST 走天下」。

### 3. 对决策者的启发
技术债在接口层会复利；早期规范省后期集成成本。

### 4. 认知升级点
HTTPS 加密传输层，不消除 Method 语义层差异。

## 五、观点辩论

### 核心观点列表（≤5）
1. RESTful 使用正确 HTTP 动词是工程规范而非形式主义。
2. 「全 POST 更安全」在 HTTPS 下不成立。
> 筛选理由：文章直接回应的争议点。

### 观点一深度辩论
**[内部项目可以全部 POST 以省沟通]**

**第一轮**
- 支持方：前后端少吵、上线快、内网无缓存问题。
- 反对方：幂等/缓存/监控/网关规则受损；新人难理解；与 HTTP 生态对抗。
- 综合方：极早期原型可妥协；对外或长生命周期 API 应遵循语义。 | 共识进度：🟢

**辩论结论**：共识表述——「短期省事的全 POST 会以集成与运维成本加倍偿还。」

### 观点二深度辩论
**[POST 比 GET 更安全故应用 POST 传一切]**

**第一轮**
- 支持方：参数不进 URL 日志。
- 反对方：HTTPS 下 GET 体虽少见但查询串同样加密；POST 破坏缓存与 CSRF 防护策略差异。
- 综合方：敏感操作用 POST+Token 合理，但不等于「只许 POST」。 | 共识进度：🟢

**辩论结论**：共识表述——「安全靠 TLS 与鉴权设计，非滥用 POST。」
